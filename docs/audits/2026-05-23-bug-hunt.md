### 1. Maithili rescue is a compound bug, not two unrelated ones

The SEO audit found Maithili was dropped from `generateStaticParams` on 2026-05-17 and lost `hreflang` for a week ŌĆö causing the click-drop. This audit's P0-4 (`getMessageFallback` returns raw key) and P0-12/P0-25 (missing `nav.*` and other regional keys) are the *second half* of that same story: even when a Maithili visitor reaches the page, the navbar renders `"allTools"`, `"financialAstrology"`, `"tropicalCompare"` as literal identifiers. ### 2. P0-2 (welcome-email bombardment) severity escalation

Per the May-20 incident memory and the audit's own framing that this is the "identical shape" of that incident, treat P0-2 as brand-damage class. It is first-priority within Batch B in the new sequence below ŌĆö not third.

### 3. P0-15 (sunrise.ts) phrasing was too soft

Original wording "works on Vercel UTC by accident, off in dev" undersells it. Reframed: **the return type of `getSunTimes()` leaks the server's local timezone into every consumer that calls `.getHours()` on the returned Date.** Vercel happens to set this to UTC today but no Next.js/Vercel contract guarantees that ŌĆö region migration or a single env var flip moves prod off-UTC. Consumers include `panchang-calc.ts`, `eclipse-compute.ts`, `parana-compute.ts`, which then feed JD math downstream. The leak is structural, not environmental.

### 4. Soft spots ŌĆö three P0 framings need nuance

- **P0-5 (Stripe metadata trust)** is a real concern but the exploit surface is narrower than the bullet implies. Webhook signing prevents replay from an outside attacker; the actual paths to abuse are (a) IDOR at `/api/checkout` allowing metadata manipulation for a different user_id, or (b) an attacker who controls a Stripe customer they own and can craft a session whose metadata names a victim user_id. Still P0 (subscription mis-assignment is high-severity) ŌĆö just not "anyone can do this."
- **P0-22 (MARANA Karaka Sthana divergence)** is partially a doctrinal disagreement, not a pure bug. Some Jyotish lineages include Rahu/Ketu in MKS; some don't. The defect is that *the same codebase* asserts both. Fix should be a deliberate authoring choice (with a written rationale), then aligned across all engines ŌĆö not auto-aligned to the longer list.
- **P0-28 (`/api/tippanni` accepts client kundali, no auth)** ŌĆö verify the public exposure before treating as exploitable. If today's surface is only authenticated client code, the attack needs an explicit unauthenticated POST attempt. The fix (auth + server-side snapshot) is correct regardless; the urgency depends on whether anyone outside the SPA can reach the route.

## P0 findings

### Security & data integrity

#### P0-1 ŌĆö `domain_readings` RLS wide open

- **File:** `supabase/migrations/012_domain_readings.sql:27-31`
- **Evidence:**
  ```sql
  CREATE POLICY "Users read own readings" ON public.domain_readings
    FOR SELECT USING (auth.uid() = user_id);
  CREATE POLICY "Service role manages readings" ON public.domain_readings
    FOR ALL USING (true) WITH CHECK (true);
  ```
- **Why it's a bug:** The second policy isn't scoped to `auth.role() = 'service_role'` like every sibling migration. PostgREST applies OR, so any signed-in user can SELECT/INSERT/UPDATE/DELETE any row ŌĆö including other users' 8-domain life scores, dasha labels, and sade-sati flags via the anon-key client.
- **Proposed fix:** Replace second policy with `USING (auth.role() = 'service_role')` matching `family_readings`/`ai_readings`.

#### P0-2 ŌĆö Service worker cross-user cache poisoning

- **File:** `public/sw.js:126,189-204`
- **Evidence:** All `/api/*` paths except `panchang` and `kundali` go through `netFirst`, which `cache.put(r, res.clone())` by URL only ŌĆö Authorization header is NOT part of the cache key.
- **Why it's a bug:** User A's `/api/user/profile` is cached; user A signs out, user B signs in. An offline or flaky request returns user A's profile JSON to user B. Same shape for `/api/subscription`, `/api/notifications`, `/api/journal`, `/api/predictions`, `/api/life-events`, `/api/family-synthesis`, `/api/brihaspati*`.
- **Proposed fix:** Skip caching for authenticated endpoints in the fetch handler, and post a message to the SW on sign-out to wipe the `CA` cache.

#### P0-3 ŌĆö Stripe webhook trusts `metadata.user_id`

- **File:** `src/app/api/webhooks/stripe/route.ts:48-73`
- **Evidence:** `checkout.session.completed` reads `session.metadata.user_id` and upserts `subscriptions` keyed on that user, with no cross-check that the session was created by that user. Brihaspati's parallel webhook (`src/app/api/brihaspati/webhook/stripe/route.ts:46-69`) does it correctly via a `brihaspati_questions` pending-row check.
- **Why it's a bug:** Anyone able to influence metadata at checkout-creation, or replay a captured webhook payload, can mint Pro/Jyotishi on any user_id.
- **Proposed fix:** Persist a `pending_checkouts` row at `/api/checkout` bound to `(session_id, user_id)`; on webhook, require a matching row before upserting.

#### P0-4 ŌĆö Stripe main webhook lacks event-id idempotency

- **File:** `src/app/api/webhooks/stripe/route.ts:47-184`
- **Evidence:** No `event.id` lookup; `customer.subscription.updated/deleted` and `invoice.payment_failed` do unconditional `.update()` on every retry. Stripe retries for 3 days and can deliver out of order.
- **Why it's a bug:** A re-played older `subscription.updated` overwrites a newer `cancel_at_period_end=true` ŌåÆ resurrects a cancelled subscription. The Brihaspati handler dedups correctly via `brihaspati_webhook_events`.
- **Proposed fix:** Insert `event.id` into a `processed_webhook_events` table with `ON CONFLICT DO NOTHING` at the top of the handler; bail with 200 if present. Use `event.created` for last-write-wins.

#### P0-5 ŌĆö Stripe webhook silently drops paid sessions when `session.subscription` is absent

- **File:** `src/app/api/webhooks/stripe/route.ts:53`
- **Evidence:** `if (userId && tier && session.subscription && session.customer)` ŌĆö when the guard fails, the function returns `{ received: true }` at line 186 with no log and no DB row.
- **Why it's a bug:** A one-time payment session, or a session needing async payment_intent follow-up, falls through silently. User paid, no subscription row, no diagnostic.
- **Proposed fix:** Log + write an audit row to `processed_webhook_events` when a paid checkout matches no handler.

#### P0-6 ŌĆö Razorpay webhook idempotency keyed on `payment_id`, with `Date.now()` fallback

- **File:** `src/app/api/brihaspati/webhook/razorpay/route.ts:25-28`
- **Evidence:** `eventId = ŌĆ” ?? \`${event.event}-${Date.now()}\`` ŌĆö replays with missing entity ID each insert as unique.
- **Why it's a bug:** Same subscription_id can legitimately have `activated`, `charged`, `cancelled` events ŌĆö keying on payment_id treats the second as a dup. The `Date.now()` fallback defeats dedup entirely when missing.
- **Proposed fix:** Key idempotency on `x-razorpay-event-id` header. Unique constraint `(provider_event_id, event_type)`.

#### P0-9 ŌĆö `getUserTier` swallows Supabase error ŌåÆ paying user demoted on transient DB blip

- **File:** `src/lib/subscription/check-access.ts:18-23`
- **Evidence:**
  ```ts
  const { data } = await supabase
    .from('subscriptions')
    .select('tier, status, current_period_end, trial_end')
    .eq('user_id', userId)
    .single();

  if (!data) {
    const result: TierResult = { tier: 'free', status: 'active' };
  ```

- **Why it's a bug:** No `error` capture. Any transient PostgREST failure ŌåÆ `data === null` ŌåÆ user silently demoted to free tier, cached for 60s. Pro user hits the paywall mid-session.
- **Proposed fix:** Destructure `{ data, error }`, log on error, do NOT cache `'free'` when the cause was an error.

#### P0-10 ŌĆö `checkAndIncrementUsage` ignores SELECT error ŌåÆ quota bypass

- **File:** `src/lib/subscription/check-access.ts:73-99`
- **Evidence:** SELECT errors ŌåÆ `data` null ŌåÆ `currentCount` defaults to 0 ŌåÆ gate passes ŌåÆ `increment_usage` runs.
- **Why it's a bug:** Free user can blow past the 2/day Brihaspati cap on DB error. Combined with the `getUserTier` bug above, a paying user can be downgraded AND a free user can bypass ŌĆö symmetrical correctness failure.
- **Proposed fix:** Fail closed on error.

#### P0-11 ŌĆö `withFeatureGate.extractUserId` swallows auth error ŌåÆ 401 turns into "anonymous free user"

- **File:** `src/lib/subscription/api-gate.ts:21,33`
- **Evidence:** `const { data } = await supabase.auth.getUser(token); return data.user?.id ?? null;` ŌĆö no `error` capture.
- **Why it's a bug:** Rate-limited / network-blipped `getUser()` returns null user ŌåÆ gate treats as anonymous free ŌåÆ logs show nothing ŌåÆ real users get downgraded mid-session.
- **Proposed fix:** Capture error, distinguish "no auth" from "auth failed" ŌĆö return 401 for the latter.

#### P0-12 ŌĆö Cron `transit-alerts` notification-insert error swallowed ŌåÆ push fires for a row that doesn't exist

- **Proposed fix:** Destructure `{ error }`; log + `continue` before sending push.

#### P0-13 ŌĆö Cron `email-alerts` notification-insert error swallowed ŌåÆ daily re-send bombardment

- **File:** `src/app/api/cron/email-alerts/route.ts:106-111, 147-152`

#### P0-14 ŌĆö Cron `domain-activations` storage failure of new reading row logged but loop still counts as "notified"

- **File:** `src/app/api/cron/domain-activations/route.ts:225-247`

### Time / date

#### P0-15 ŌĆö `getSunTimes()` returns Dates whose meaning depends on server timezone

- **File:** `src/lib/astronomy/sunrise.ts:78-88`
- **Evidence:**
  ```ts
  // `minutes` are local-time minutes. Date stores local h:m:s in the
  // machine's timezone. Consumers must use getHours()/getMinutes() ...
  return new Date(year, month - 1, day, h, m, s);
  ```
  Comment admits the leak. Consumers (`panchang-calc.ts:1139-1140`, `eclipse-compute.ts:127-128`) call `.getHours()`.
- **Why it's a bug:** Lesson L violation propagated through panchang transitions, Rahu Kaal, choghadiya boundaries, eclipse phases. Works on Vercel UTC by accident; off-by-server-offset hours in dev. Extreme latitudes (`minutes < 0` or `>= 1440`) silently roll the calendar day backward.
- **Proposed fix:** Return decimal local hours (or `{date, hour, minute}` POJO). Never expose a Date whose accessors depend on local tz.

#### P0-16 ŌĆö Birth weekday for Gulika/Mandi: `utHour < sunriseUT` not normalised when sunset crosses midnight UT

- **File:** `src/lib/ephem/kundali-calc.ts:995-996`
- **Evidence:** `const birthDateObj = new Date(Date.UTC(year, month - 1, day)); const weekday = birthDateObj.getUTCDay();` ŌĆö fine. But the `isDayBirth = utHour >= sunriseUT && utHour < sunsetUT` check doesn't add 24 when `sunsetUT < sunriseUT` (west-of-UTC births).
- **Why it's a bug:** Night births in PST get classified as day births ŌåÆ wrong Gulika segment ŌåÆ wrong Gulika/Mandi positions ŌåÆ wrong remedy + dignity calc.
- **Proposed fix:** Normalise `sunsetUT += 24` when `< sunriseUT`; normalise `utHour += 24` if `< sunriseUT - 12`.

#### P0-17 ŌĆö Eclipse compute inherits sunrise.ts bug

- **File:** `src/lib/calendar/eclipse-compute.ts:127-128`
- **Evidence:** `srH = times.sunrise.getHours() + ŌĆ”/60`
- **Why it's a bug:** Off-by-server-tz-offset ŌåÆ eclipse visibility / sutak windows misreported in dev.
- **Proposed fix:** Switch caller to a UT-hours API; pass `tzOffset` explicitly.

#### P0-18 ŌĆö Ashtamangala "current time" double-applies tz offset

- **File:** `src/lib/prashna/ashtamangala.ts:133-141`
- **Evidence:**
  ```ts
  const utcMs = now.getTime() + now.getTimezoneOffset() * 60000;
  const localMs = utcMs + tzOffset * 3600000;
  const local = new Date(localMs);
  ... local.getFullYear(); local.getHours();
  ```
- **Why it's a bug:** `now.getTime()` is already UTC; adding `getTimezoneOffset()` shifts by server tz. Then `local.getHours()` adds another local-tz shift. Vercel UTC by accident works; non-UTC dev wrong by 1-2h ŌåÆ wrong tithi ŌåÆ wrong prashna verdict.
- **Proposed fix:** Use `Intl.DateTimeFormat(..., { timeZone })` to format components directly; never round-trip via Date local accessors.

#### P0-19 ŌĆö Solar-return `jdToDateObj` adds tzOffset then re-constructs with `Date.UTC`

- **File:** `src/lib/varshaphal/solar-return.ts:89-111`
- **Evidence:** `hourFrac = (dayFrac - day) * 24 + tzOffset; return new Date(Date.UTC(year, month-1, day, hour, min));`
- **Why it's a bug:** Constructs a Date whose UTC components equal local wall-clock. `toISOString()` returns local labeled `Z`. `srWeekday` (read with `getUTCDay()` in `varshaphal/index.ts:52`) can be the wrong day. `year/month/day` not normalised when `tzOffset` pushes hour overflow.
- **Proposed fix:** Construct from UT components, then convert to local separately.

#### P0-20 ŌĆö `findTithiDate` / `findEkadashiDate` use local-tz Date math near DST

- **File:** `src/lib/calendar/festivals.ts:53-78, 134-198`
- **Evidence:** `new Date(year, month-1, 1).setDate(getDate() - 15)` etc. Then `(d1-d0)/86400000 <= 1` for Dwi-Ekadashi check.
- **Why it's a bug:** DST adds ┬▒1h per `setDate`; over 15 days drifts can flip the comparison.
- **Proposed fix:** Use `Date.UTC` and `getUTC*` throughout.

## Cross-cutting themes

1. **Lesson L is pervasive.** `new Date(year, m-1, d, h, m)` with later `getHours()/getMinutes()/getMonth()` accessors lives in ~30 files. Most happen to work on Vercel UTC. Any non-UTC dev machine, or any user near a calendar-day boundary in their own TZ, hits a wrong value. **The biggest single risk:** `getSunTimes()` is the root contagion ŌĆö its return type leaks server tz into panchang transitions, Rahu Kaal, choghadiya boundaries, eclipse phases. Fix the contract there and ~10 P0/P1 findings collapse.

2. **Lesson Q is realised in 5 distinct ways now.** Two cases of actual divergence (`MARANA_KARAKA`, Rahu-Saturn friendship); three pre-drift duplicates (SIGN_LORD, BENEFICS/MALEFICS, EXALTATION). One fix-and-import campaign moves all of `src/lib/constants/` to be the single source of truth.

3. **Webhook idempotency is split-brain.** Brihaspati's webhook gets it right (event-id table, atomic insert with unique constraint). The main Stripe + Razorpay webhooks do not. Highest-leverage fix in this audit: port the Brihaspati pattern to both.

4. **Subscription gating's error-blindness is symmetric.** `getUserTier` demotes paying users (over-strict on DB blip); `checkAndIncrementUsage` lets free users blow past quota (under-strict on DB blip). Fixing one without the other leaves the bill skewed in opposite directions. Fix together.

5. **Cron writes are send-first, mark-after across the board.** `email-alerts`, `transit-alerts`, `onboarding-drip`, `domain-activations` all share the same shape ŌĆö send the user-visible action, then write the dedup row, with no transaction/outbox. Any single DB blip ŌåÆ repeat-send tomorrow. Same shape as the May-20 incident. A single shared outbox helper would close all four.

6. **i18n missing-key surfacing is broken at the root.** `getMessageFallback` returning the leaf key turns every missing translation into raw camelCase identifier in the UI. Combined with 600+ missing keys in regional locales (caesarean-muhurta + learn pages + nav.*), this is visible to Maithili users today. The runtime fallback fix is one-line; backfilling values can stay as a follow-up via `/translate-locale`.

7. **`String(err)` PII leakage to clients in 9+ routes.** Most older routes were already hardened. The 9 listed above (calendar/export, almanac, muhurta-scan, caesarean-scan, festival-compare, tithi-table, cron/daily-panchang, cron/social-post, social/youtube, social/instagram) are the remaining holes. Generic message + tagged server log fixes them all.

8. **Rate-limit infrastructure is per-Lambda.** Six in-memory `Map` limiters across AI endpoints; Vercel runs many concurrent containers; an attacker (or a bursty user) gets N ├Ś the documented limit. KV/Upstash migration is the proper fix; this is acknowledged in the code as "good enough before Redis."
