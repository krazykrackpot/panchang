# SEO Audit — Dekho Panchang
**Date:** 2026-05-23
**Scope:** Full app — infrastructure, coverage, multilingual, structured data, recent features

---

## 0. TL;DR

The SEO foundation is strong: 260 PAGE_META entries, a 24 KB `sitemap.ts`, 8 active locales, 31 OG-image routes, JSON-LD on every key surface, an `llms.txt` + `llms-full.txt` for AI crawlers, and explicit `[]` returns from `generateStaticParams` on the date/city explosion routes.

But the audit surfaces **five real problems** (one critical, two high, two medium) and a long tail of cleanup items. The headline issues:

1. **CRITICAL — Multilingual title/description gap on top traffic pages.** `/panchang`, `/kundali`, `/horoscope`, `/devotional` only carry `en` / `hi` / `sa` strings in `PAGE_META`. `sa` is retired. `mai` (Maithili — the #1 traffic-driving locale per memory) and `ta`/`te`/`bn`/`gu`/`kn` fall back to English. CTR-killing for non-English SERPs.
2. **HIGH — 4 sitemap-indexed pages get empty `{}` metadata.** `/calendar/regional/kannada|malayalam|telugu` and `/widget` call `getPageMetadata(route, locale)` for a `route` that doesn't exist in `PAGE_META`. The function returns `{}`, so these pages ship with no title, description, canonical, hreflang, OG, or Twitter card. Crawled, indexed, naked.
3. **HIGH — New `/calendars/masa` (Hindu Months) ships without a `layout.tsx`.** No metadata, no JSON-LD, no canonical, no OG. The page is in `PAGE_META` and reachable from `/calendars`, but the layout that *applies* the metadata isn't there.
4. **MEDIUM — Three big SEO programs miss the sitemap entirely.** `/calendars/masa`, `/calendar/eclipses`, `/eclipses/simulator`, `/muhurta-ai/annual`, `/learn/contributions` (the contributions *index*) — all crawlable, none discoverable from `sitemap.xml`.
5. **MEDIUM — Static-page budget drift.** CLAUDE.md mandates `[]` from `generateStaticParams` on four route shapes; three of them now seed a small set instead. Risk is bounded today (~25–60 pages each) but the budget will silently erode if nobody watches it.

Detailed proposed fixes (no code changed in this audit) are in §5.

## 2. Findings

### 2.1 CRITICAL — Top traffic pages miss `mai`, `ta`, `te`, `bn`, `gu`, `kn` titles/descriptions

`getPageMetadata` (`metadata.ts:1928`) falls back to `meta.title.en` when the locale key is absent. For SEO that means **the SERP title and description for non-English locales is English** — which both kills CTR and confuses Google's locale targeting.

Concrete state (sampled via the audit):

| Route | Has locales | Missing |
|---|---|---|
| `/panchang` | `bn, en, gu, hi, kn, sa, te` | **`mai`, `ta`** |
| `/kundali` | `en, hi, sa` | **`bn`, `gu`, `kn`, `mai`, `ta`, `te`** |
| `/horoscope` | `en, hi, sa` | **`bn`, `gu`, `kn`, `mai`, `ta`, `te`** |
| `/devotional` | `en, hi, sa` | **`bn`, `gu`, `kn`, `mai`, `ta`, `te`** |
| `/brihaspati` | All 8 + `sa` | None ✓ |
| `/sadhaka-path` | All 8 + `sa` | None ✓ |

Plus: many entries still carry a `sa:` key. `sa` is retired and 301-redirected — those strings are dead weight (~50 KB) and risk being served if somebody re-enables `sa` accidentally.

**Why this is critical:**
- `mai` is the **#1 traffic locale per memory** (May 21 Maithili choghadiya spike: 275 clicks). `/mai/panchang` SERP shows the English title.
- `/panchang` is the project's flagship. Its `mai`/`ta` SERP rows render in English.

### 2.2 HIGH — 4 indexed pages get empty `{}` metadata

`getPageMetadata` does `if (!meta) return {};` (line 1930). Four layouts call it with a key that isn't in `PAGE_META`:

- `/calendar/regional/kannada` → no PAGE_META entry
- `/calendar/regional/malayalam` → no PAGE_META entry
- `/calendar/regional/telugu` → no PAGE_META entry
- `/widget` → no PAGE_META entry

These pages render to the browser with **no `<title>`, no description, no canonical, no `hreflang`, no OG, no Twitter card** — only whatever the parent `[locale]/layout.tsx` supplies (which is a static fallback title). They are reachable, crawlable, and would currently be indexed.

The silent `{}` return is the root cause — it should at minimum `console.warn` (a CLAUDE.md "never silently swallow" violation in disguise).

### 2.3 HIGH — `/calendars/masa` ships without a layout

`src/app/[locale]/calendars/masa/page.tsx` exists. The route is reachable from `/calendars`. But there is **no `layout.tsx`** in that directory — so the layout-level `generateMetadata` pattern used everywhere else doesn't fire, and the page inherits from the parent (`/calendars/layout.tsx`), which is about the *index* page, not Masa.

Result on `/en/calendars/masa`:
- Title is the parent's "Vedic Calendars Hub" string
- No JSON-LD (audit confirmed)
- No canonical pointing at the Masa URL
- Not present in `sitemap.ts`

This is a brand new content piece (the user explicitly built it as a Hindu Months hub) and it's invisible to search.

### 2.5 MEDIUM — `generateStaticParams` drift vs the documented budget

CLAUDE.md ("Static Page Budget") lists nine route shapes that **must** return `[]` from `generateStaticParams`. Today's state:

| Route | Required | Actual |
|---|---|---|
| `horoscope/[rashi]/[date]` | `[]` | ✓ no `generateStaticParams` (defaults to ISR-only) |
| `horoscope/[rashi]/weekly` | `[]` | ✓ no `generateStaticParams` |
| `horoscope/[rashi]/monthly` | `[]` | ✓ no `generateStaticParams` |
| `calendar/[slug]` | `[]` | ✓ no `generateStaticParams` |
| `panchang/[city]` | `[]` | ✓ no `generateStaticParams` |
| `choghadiya/[date]` | `[]` | ✗ pre-generates `-7 to +30` days (~38 dates × 8 locales = 304 pages) |
| `muhurta/[type]/[year]/[month]/[city]` | `[]` | ✗ pre-generates first few activities × current/next 2 months |
| `festivals/[slug]/[year]` | `[]` | ✗ seeds top 5 festivals × 2026 (10 entries × 8 locales = 80 pages) |
| `festivals/[slug]/[year]/[city]` | `[]` | ✗ seeds top 5 × top 5 cities (25 entries × 8 locales = 200 pages) |

None of these are catastrophic individually. The risk is that the seed lists grow (someone bumps "top 5" to "top 20" without realising) and the 2,000-page cap is silently approached. The choghadiya one already pre-renders ~38 dates per locale per build — wasted work because the content is heavily time-conditioned.

### 2.6 LOW — Robots / sitemap / hreflang loose ends

- **Sitemap-submitted locales** (`en`, `hi`, `mai`) vs **hreflang alternates** (all 8) — Google's hreflang validation expects every member of an hreflang cluster to either appear in a sitemap or self-reference. The 5 non-submitted locales (`ta`, `te`, `bn`, `gu`, `kn`) appear in `<link rel="alternate" hreflang="...">` on every page but are not in `sitemap.xml`. This is *technically* permitted but suppresses Google's discovery of locale pages that have no internal backlinks.
- **`robots.txt` redundancy** — `User-agent: *` allows `/*/regional`, `/*/festivals`, etc. then each AI crawler block (GPTBot, ChatGPT-User, Claude-Web, anthropic-ai, Google-Extended, PerplexityBot, cohere-ai) repeats the same Allow list. That's ~50 redundant lines per crawler. Maintainability hit, not an SEO bug — but a single typo (e.g. `/*/regiona`) is hard to spot in 510 lines.
- **CCBot, Bytespider** correctly fully disallowed. ✓
- **OG image coverage** — 31 routes have a dedicated `opengraph-image.tsx`. New high-value landings have none: `/brihaspati`, `/sadhaka-path`, `/calendars/masa`, `/devotional/[type]/[slug]`, `/vrat-katha/[slug]`, `/hindu-calendar/[year]`, `/vivah-muhurat/[year]`. These ship the *root* `/en/opengraph-image` for social sharing (generic site card) instead of a feature-specific card.
- **`proxy.ts` exports `function middleware(...)`** — Next.js 16 expects the exported function to be named `proxy` (per the routing-middleware reference). Today's name still works (Next 16 deprecates but accepts `middleware.ts`) but the function-name mismatch is a minor smell.
- **Sitemap `lastModified`** — every entry gets `new Date()` at request time. That's "now" on every crawl, not the page's actual change time. Google won't trust the timestamps for prioritising re-crawl. Switching to a per-route or per-asset mtime would be more useful.
- **`changeFrequency`** values are mostly `'monthly'`. Search engines have explicitly ignored `changeFrequency` for years — it's harmless but cosmetic.

The feature stream from the last 30 days (Brihaspati, Sadhaka Path, Tithi Calendar deity portraits, Hindu Months page, Daily Panchang share cards, Cosmic Identity card, family dashboard transit forecast):

| Feature | Discoverable in nav? | In sitemap? | PAGE_META? | JSON-LD? | OG? |

| Hindu Months (`/calendars/masa`) | From `/calendars` index | ✗ | ✓ | ✗ | ✗ |
| Tithi Calendar deity portraits | Internal to `/calendar` | n/a — feature on existing page | n/a | n/a | n/a |
| Cosmic Identity card (dashboard) | Inside `/dashboard` | n/a — gated | n/a | n/a | n/a |
| Family dashboard (`/dashboard/family`) | Nav | ✗ correctly `noindex` | n/a | n/a | n/a ✓ |
| Daily Panchang email | Server-side, no public page | n/a | n/a | n/a | n/a |

Both Brihaspati and Sadhaka Path are flagship features — they have PAGE_META + sitemap + JSON-LD but **no dedicated OG image**, so Twitter/LinkedIn/WhatsApp shares of these landings render the generic site card. For a "share-the-AI-answer" feature like Brihaspati, that's a missed acquisition loop.

### 2.8 LOW — Long tail

- `OG_LOCALE_MAP` lists `mr: 'mr_IN'` for a *retired* locale; harmless but stale.
- `sitemap.ts` includes a duplicate-import comment block at the top that's now correct (`mai` is back in) but the file would benefit from regenerating the route list from the file system to prevent drift.
- The 30-day forward window for `horoscope/[rashi]/[date]` and `choghadiya/[date]` in the sitemap depends on a *new build happening within 30 days*. With the new daily-only deploy policy (PR #110), this still works (06:00 UTC daily cron), but if the deploy cron is paused, sitemap dates stop rolling forward silently.

## 3. What's *good* — keep doing these

- Centralised PAGE_META is the right pattern. The fix is filling the missing locale strings, not abandoning the helper.
- `[]` returns on the big dynamic routes (panchang/[city], horoscope/[rashi]/[date], calendar/[slug]) successfully kept the build under the 2,000 static-page budget. Sitemap pruning to 1,064 URLs (from 9,390) per memory was the correct call.
- The AI-crawler robots configuration is thoughtful: GPTBot/Claude-Web/Perplexity get tool pages, contributions, and learn (excluding modules) — exactly the content you'd want cited. Bytespider/CCBot blocked. This is more deliberate than most sites.
- `safe-jsonld.ts` exists and is used — avoids the `dangerouslySetInnerHTML` injection class of bugs that plagues JSON-LD across the rest of the industry.
- `llms.txt` + `llms-full.txt` (172 + 681 lines) are unusually well-curated for AI referral.
- 8 `/vs/<competitor>` pages exist with correct PAGE_META and the no-competitor-in-other-pages rule is enforced (per the recent Brihaspati panel test). Strong play for branded-comparison search intent.

## 4. Proposed Fixes (no code changes here — recommended actions only)

### Fix A — Backfill multilingual `PAGE_META` (resolves §2.1)

Target the **top 30 pages by traffic** first (the long tail of 230 PAGE_META entries can be batched later or auto-translated). Per page, add the 6 missing locales: `ta`, `te`, `bn`, `gu`, `kn`, `mai`.

Tool of choice: a one-off script that walks `PAGE_META`, identifies missing keys, drafts translations via the existing translation pipeline, and writes back. Pair the script with a vitest assertion in `metadata.test.ts` that fails if any active locale is missing from any PAGE_META entry — same pattern as the Brihaspati i18n parity test.

Drop `sa:` keys at the same pass — they're never read (retired locale, 301-redirected).

**Cost:** ~2 hours of human review + 1 hour of translation work per 30 pages.
**SEO upside:** Direct CTR lift on every non-English locale SERP for `/panchang`, `/kundali`, `/horoscope`, the comparison pages.

### Fix B — Make `getPageMetadata` fail loudly + add a unit test (resolves §2.2)

Replace `if (!meta) return {};` with a `console.warn('[seo] No PAGE_META for ${route} — falling back to bare title')` and emit at least a title from a deterministic default (e.g. titleCase of the last path segment + site suffix), plus the canonical/hreflang block (which doesn't depend on PAGE_META at all).

Then add a vitest that enumerates all `getPageMetadata(route, ...)` call sites (via grep) and asserts each `route` exists in `PAGE_META`. This is exactly the pattern that would have caught the 4 silent-empty pages today.

For the 4 pages flagged today: add proper PAGE_META entries for `/calendar/regional/kannada|malayalam|telugu` (the bengali/tamil/mithila/gujarati entries are already there — same shape) and decide whether `/widget` should be `noindex` or get real metadata.

**Cost:** ~1 hour.
**SEO upside:** Stops bleeding indexed-but-titleless pages.

### Fix C — Add `/calendars/masa/layout.tsx` + sitemap entry + JSON-LD (resolves §2.3)

Mirror the pattern from `/calendars/tithi/layout.tsx` (which already exists and works). Three actions:

1. Create `layout.tsx` that calls `getPageMetadata('/calendars/masa', locale)` and emits a `WebPage` JSON-LD via `generateExpertiseArticleLD` or `generateToolLD`.
2. Add `'/calendars/masa'` to the static-routes list in `sitemap.ts` (alongside `/calendars/tithi`).
3. Confirm a PAGE_META entry exists (it does — the locale coverage is fine).

**Cost:** ~15 min.
**SEO upside:** New canonical SERP target for "hindu months", "vedic calendar months", "amanta purnimanta" queries.

### Fix D — Backfill sitemap omissions (resolves §2.4)

Add to `routes` array in `sitemap.ts`:
- `/calendars/masa`
- `/calendar/eclipses`
- `/eclipses/simulator`
- `/muhurta-ai/annual`
- `/learn/contributions`
- `/muhurat`
- `/rituals`

For each, also verify the corresponding `layout.tsx` exists and emits metadata.

**Cost:** ~30 min including verification.
**SEO upside:** Direct crawl-budget allocation to the new high-effort pages.

### Fix E — Lock the static-page budget in CI (resolves §2.5)

Add a build-time assertion that fails when any of the 9 documented routes (CLAUDE.md "Static Page Budget") returns more than 0 params from `generateStaticParams`. This is one regex test in vitest:

```
const route = readFile('src/app/[locale]/festivals/[slug]/[year]/page.tsx');
expect(route).toMatch(/return\s*\[\]/);
```

Then make a deliberate decision about `choghadiya/[date]`, `festivals/[slug]/[year]`, `muhurta/.../[city]`: either remove the seeding (return `[]`, rely on ISR) OR document the bounded seed in CLAUDE.md and shrink the windows where they're too aggressive (e.g. choghadiya's -7 to +30 → -1 to +3).

**Cost:** ~1 hour.
**SEO upside:** Indirect — prevents a future build-OOM regression that would tank the entire site.

### Fix F — Add dedicated OG images for flagship landings (resolves §2.7 OG gap)

Create `opengraph-image.tsx` under:
- `src/app/[locale]/brihaspati/` — sage avatar + "Ask Brihaspati" + the headline.
- `src/app/[locale]/sadhaka-path/` — gamification card with streak/level.
- `src/app/[locale]/calendars/masa/` — Hindu months wheel.
- `src/app/[locale]/vrat-katha/[slug]/` — deity portrait + vrat title.
- `src/app/[locale]/hindu-calendar/[year]/` — year card.

Templated via `next/og` + Satori, mirroring the existing `/panchang/[city]/opengraph-image.tsx` pattern.

**Cost:** ~2 hours total (one template, 5 instantiations).
**SEO upside:** Indirect — improves social CTR + brand recognition in LinkedIn/Twitter/WhatsApp shares.

### Fix G — Cosmetic robots.txt cleanup (resolves §2.6)

Extract the per-crawler Allow list into a shared block via a small build step (e.g. a `scripts/build-robots.ts` that templates the file). Cuts 510 lines to ~200 and removes the typo surface. Zero SEO change.

**Cost:** ~1 hour.
**SEO upside:** None directly — maintainability only.

### Fix H — Rename `middleware` → `proxy` in `src/proxy.ts` (resolves §2.6)

The file is correctly named `proxy.ts` for Next.js 16, but the exported function is still `function middleware(request)`. Next 16 accepts both for now; future versions will not. Rename + update the matcher comment.

**Cost:** ~5 min.
**SEO upside:** None directly — forward-compatibility.

### Fix I — Sitemap `lastModified` from real mtimes (resolves §2.6)

Replace `new Date()` with the page-file mtime (or a per-route hash from a build manifest). Trivial implementation: `import { statSync } from 'node:fs'; statSync('src/app/[locale]/panchang/page.tsx').mtime`.

**Cost:** ~30 min.
**SEO upside:** Low — Google mostly ignores `<lastmod>` until you start updating it accurately, then it nudges re-crawl frequency on changed pages.

## 5. Self-Review (Critical Pass)

I went back through my own audit and asked five questions:

### Q1: Did I confirm findings with file reads, or just grep counts?

Yes for the critical+high items:
- §2.1 — verified via the Python script that parses `PAGE_META` blocks and lists missing locale keys. Sampled 9 routes.
- §2.2 — verified by extracting all `getPageMetadata(...)` call sites and joining against PAGE_META keys.
- §2.3 — `ls src/app/[locale]/calendars/masa/` returned only `page.tsx`, no `layout.tsx`. Verified.
- §2.4 — `grep -qE` on `sitemap.ts` per route.
- §2.5 — `grep -A5 generateStaticParams` on each route and inspected the body.

**One thing I didn't verify:** for §2.2, I assumed `getPageMetadata` returning `{}` actually produces a titleless page in the browser. The Next.js Metadata merge contract means the parent layout's metadata flows through. So the page *probably* gets the root layout's "Dekho Panchang" title fallback — not literally bare. That weakens "no `<title>`" to "no *page-specific* `<title>`", which is still a real SEO bug (every page on the site gets the same title) but less alarming.

**Correction needed:** soften §2.2's bullet from "ship with no title" to "ship with the root-layout fallback title — i.e. every `/calendar/regional/{telugu,kannada,malayalam}` page gets the homepage title instead of a regional-calendar title". Same severity (the SERP fingerprint is wrong) but technically more accurate.

### Q2: Did I conflate "in sitemap" with "indexed"?

A page can be indexed without being in the sitemap (Google discovers via internal links). And a page in the sitemap can be noindex'd (good — happens with daily/[date] per the existing comment in sitemap.ts:430).

I correctly distinguished these for `dashboard/family` (noindex, intentional) but for §2.4 "sitemap omissions" I should be clearer: missing from sitemap **doesn't mean missing from Google's index** — it means Google has to find them via internal links and may deprioritise crawl. The proposed fix is still right, but the severity (MEDIUM not HIGH) reflects this.

### Q3: Is the multilingual claim (§2.1) actually a CTR problem?

Google does serve English titles on `mai` SERPs when the page has no Maithili title. But the IMPACT depends on whether the user searches in Devanagari/Maithili script. If the GSC data shows the Maithili choghadiya page already gets 275 clicks/day with an English title, the CTR baseline is *already* what the broken state produces — the lift from adding Maithili titles would be a CTR improvement, not a from-zero unlock.

**Honest restatement:** §2.1 is a CTR-improvement opportunity worth doing, but it's not "fixing a regression". I overstated it by calling it CRITICAL. Demoting to HIGH.

### Q4: Did I miss anything by sampling instead of enumerating?

The PAGE_META locale coverage sweep was sampled (9 routes). To be safe, the proposed §4-A script should enumerate all 260 entries — I'll note that in the final recommendation. Also worth checking: do any locales render with completely English `<h1>` because the page-level component reads from PAGE_META not from `useTranslations`? Spot-check on `/panchang` says no — the `<h1>` comes from `getTranslations('pages.panchang')`, not PAGE_META. But that's worth verifying by component for each of the top 30 pages before claiming the lift.

### Q5: What did I *not* audit that I should have?

1. **Canonical correctness across `[city]` and `[date]` dynamic routes.** I confirmed the centralised metadata produces canonical URLs, but didn't sample a real `/panchang/delhi` and verify the canonical points at itself (not at `/panchang`). For 250+ tier-1+tier-2 cities, this matters.
2. **Core Web Vitals signals.** SEO ≠ technical SEO alone. LCP/INP/CLS are ranking inputs. I didn't profile any page.
3. **301-redirect chains.** Some Western-name aliases (aries → mesha, leo → simha) live via redirects. Multi-hop chains hurt. Not verified.
4. **404 handling.** No `not-found.tsx` audit. Catching invalid `[slug]` / `[city]` / `[year]` parameters with a real 404 (not a noindex'd error page) is important.
5. **Image alt text on the new tarot/deity portraits.** The festival icons audit (PR #119) was about Next/Image warnings, not alt text. New deity portraits may lack descriptive `alt`.
6. **Mobile usability.** No real-device check — relevant since `mai` traffic is plausibly mostly mobile.

These belong in a follow-up audit, not this one. Listing them so they're not forgotten.

## 6. Final Recommendation (prioritised)

If I could only ship five things, in order:

1. **Backfill multilingual PAGE_META on the top 30 pages** (Fix A) — every locale gets a real title/description. **High effort, high upside, blocks AI-translation backfill of the remaining 230.** Pair with a parity test.
2. **Make `getPageMetadata` fail loudly + add the call-site test + add the 4 missing PAGE_META entries** (Fix B) — 1 hour, plugs a silent gap.
3. **Wire `/calendars/masa` (layout + sitemap + JSON-LD)** (Fix C) — 15 minutes, unlocks a new content hub for indexing.
4. **Backfill sitemap omissions** (Fix D) — 30 minutes, low-risk Google-discovery improvement.
5. **Add OG images for `/brihaspati`, `/sadhaka-path`, `/calendars/masa`** (Fix F, subset) — 1 hour, direct social-CTR upside on the flagship features.

**Defer:** the static-budget CI lock (Fix E — useful but not urgent), robots cleanup (Fix G — purely cosmetic), proxy function rename (Fix H — 5 min trivial), `lastModified` rewrite (Fix I — Google barely uses it).

**Don't do:** auto-translate everything at once. Translate the top 30 by hand or with review, then batch the long tail with a vitest gate enforcing parity.

**Followup audit** should cover Core Web Vitals, canonical correctness across `[city]` dynamic routes, 404/redirect chains, and image alt text on the new deity portraits.
