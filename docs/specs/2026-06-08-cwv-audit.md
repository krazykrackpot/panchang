# Core Web Vitals audit — 2026-06-08

Audit of CWV impact after the SEO-recovery work (Phase A + Phase B + city
cut + date-content Pass 1-4 + chrome density + city descriptors +
nakshatra-pada extras + festival year-context + weekly/monthly thicken).
Closes audit item **B** from `docs/specs/2026-06-08-seo-audit-followups.md`.

## TL;DR

CWV is healthy. The recovery work added ~6-7 large content modules but
**all of them stay on the server bundle** — zero client-bundle impact.
Pre-existing CWV setup (image AVIF/WebP, `display: swap` fonts,
`optimizePackageImports`, 30 `next/dynamic` usages) is already strong.
No critical issues found.

| Module | Size | Imported by | Server / client |

| `tithi-observances.json` | 729KB | `TodaySignificanceSection.tsx` (server) | Server-only |

Verification: every consumer is a server component (no `'use client'`),
every `import` chain that touches a JSON terminates in a server-rendered
template. JSON data parses once per Node process at module load
(memoised by V8) — no per-request parse cost, no client transfer.

## Rendered HTML size

After the recovery work, date-keyed page bodies grew by an estimated
1-2KB gzipped per locale (one tithi-observance paragraph + weekday
paragraph + 5-item festival list + per-city descriptor on /panchang/<city>
+ pada-extras on /learn/nakshatra-pada/*). Modest growth for substantial
content gain.

LCP impact: negligible. Most growth is text after the H1; the H1 + first
above-the-fold text still paint first.

## Recommendations (not blocking, not in this PR)

1. **Continuous monitoring.** Vercel Speed Insights is already wired
   (per memory). Watch the CWV percentiles for the 4 hottest templates:
   `/`, `/panchang`, `/choghadiya/<date>`, `/panchang/<city>`. Set
   alerts if LCP p75 crosses 2.5s or INP p75 crosses 200ms.
2. **Bundle-size budget guard.** `@next/bundle-analyzer` is installed.
   Consider wiring a CI gate that fails the build if any per-route
   client JS exceeds a budget (e.g. 200KB gzipped). Not in this PR
   because budgets need empirical tuning.
3. **HoroscopeClient lazy-load.** A theoretical micro-win — defer the
   ~30KB JS to after first paint via a wrapper-client-component pattern
   (server-component pages can't directly use `next/dynamic({ssr:false})`).
   Marginal CWV gain, adds complexity. Defer until empirical data shows
   the bundle as a real bottleneck.
4. **Heavy-image audit.** Once Vercel Speed Insights flags any specific
   LCP element as an image, run a focused image-optimization pass.

## What was NOT changed in this PR

No code changes. The audit verified everything stays within CWV-friendly
